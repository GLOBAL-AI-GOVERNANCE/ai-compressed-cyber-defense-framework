## Claim Review Rules

Before publishing a new claim:

1. Identify whether the claim is factual, interpretive, predictive, or recommended practice.
2. Attach at least one source for factual claims.
3. Mark predictive claims as forecasts or risk scenarios, not facts.
4. Avoid attributing authorship, endorsement, or affiliation unless permission is explicit.
5. Remove offensive procedural detail unless it is necessary for defensive architecture understanding.
6. Prefer bounded language: "can," "could," "may," "should," and "in controlled contexts" where appropriate.

## Public-Safe Language Standard

Preferred:
- AI-compressed cyber threats
- Frontier AI capabilities in vulnerability discovery
- Controlled security research context
- Defender-side remediation and architecture hardening
- Structural security, bounded compromise, mission resilience

Avoid:
- Claims of guaranteed security
- Claims that a named model has been misused unless publicly verified
- Target-specific exploit procedures
- Instructions for unauthorized access
- Unverified affiliation or endorsement language

## Use Rule

These sources are used for evidence grounding only. Specific vendor model names must not be used as project branding, endorsement language, or a permanent threat taxonomy.

Future releases should re-check fast-moving AI capability and model-availability claims before publication.
